# 対戦機能に関する議論メモ (2026-03-29)

## やりたいこと

同じシード値でAさんとBさんが同時に遊んで、終わった結果（スコア）で競う対戦機能。
MCSR Ranked (https://mcsrranked.com/) のようなシステムをイメージ。

## MCSR Rankedの仕組み（調査結果）

- Minecraftサーバーは使っていない
- 各プレイヤーは自分のPCでシングルプレイ
- クライアントMODがバックエンドAPI（Webサーバー）と通信
- APIが同じシード＋同じRNG設定を両者に配布 → 同一ワールドが生成される
- 進捗やスコアはAPIに送信して比較

```
[AさんのPC]                    [バックエンドAPI]                [BさんのPC]
 クライアントMOD  ── HTTP ──→  マッチング・シード配布  ←── HTTP ── クライアントMOD
 ローカルで                     スコア記録・ELO計算           ローカルで
 シングルプレイ                                              シングルプレイ
```

RTAは「同じシードで各自が独立にプレイ」なので同じワールドにいる必要がない。
lightgame100も同じ構造で成立する（同じシード＋同じ座標範囲なら地下構造が同一）。

## lightgame100への適用

### 既存コードへの影響

- ゲームロジック（スナップショット、スコア計算、タイマー等）→ そのまま使える
- 既存コードはサーバーサイドで動くが、シングルプレイでも内蔵サーバーで動くので問題なし
- 「クライアントMODに書き換える」必要はなく、クライアント側の処理を追加するイメージ

### 新規で必要なもの

- バックエンドAPIサーバー（マッチング、シード＋座標配布、スコア記録・比較）
- MODからAPIへのHTTP通信コード（クライアント側処理）

### 実現レベルの段階

**最小構成（運用でカバー）：**
- シード値と座標を手動で共有して各自ワールド作成
- スコアをファイル出力 or クリップボードコピー → Discordで共有

**もう少しちゃんとやる：**
- 簡易WebサーバーでAPIを立てる
- MODがAPIからシード＋座標＋開始信号を受信
- MODがスコアをAPIに送信 → ランキング表示

## 検討したが不採用の案

### サーバー型（専用サーバーにプレイヤーが接続）

- 1台のサーバーでプレイヤーごとに別ディメンション（ワールド）を作る案
- GameManagerのstatic状態のインスタンス化、WorldBorderの代替、ロビー/マッチング機能など大規模な変更が必要
- Fantasyなどのディメンション管理ライブラリが必要
- MCSR Ranked型のほうがシンプルなので不採用
